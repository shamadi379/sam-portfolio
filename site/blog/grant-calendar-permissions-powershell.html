Grant Calendar Permissions via PowerShell (Exchange Online / Microsoft 365)

Published: October 6, 2025 — Sam Adhikari

This short guide shows how to grant calendar permissions using PowerShell for Microsoft 365 (Exchange Online) and how to do it safely at scale. I’ll cover: connecting to Exchange Online, granting permissions to a single mailbox, granting to many mailboxes, verifying the results, and removing permissions.

When to use this

  • Allow an assistant or service account to view or manage a user’s calendar.
  • Set consistent default permissions across many mailboxes in bulk.
  • Automate onboarding/offboarding calendar access.

Important safety notes

  • Always test scripts in a non-production tenant or one test user.
  • Least privilege: grant only the permission level required (Reviewer > Editor > Owner).
  • Logging: keep a record of changes and who requested them.

1) Connect to Exchange Online (recommended module)

Install-Module -Name ExchangeOnlineManagement -Force
Import-Module ExchangeOnlineManagement
Connect-ExchangeOnline -UserPrincipalName admin@yourtenant.onmicrosoft.com -ShowProgress $true

2) Grant calendar permission for a single mailbox

Example: grant the user assistant@contoso.com Reviewer access to alice@contoso.com calendar.

Add-MailboxFolderPermission -Identity "alice@contoso.com:\Calendar" -User "assistant@contoso.com" -AccessRights Reviewer

3) Grant permission to multiple mailboxes (CSV)

Create a CSV file with a column Mailbox listing mailbox addresses, and run:

$csv = Import-Csv .\mailboxes.csv
foreach ($row in $csv) {
  $mb = $row.Mailbox
  Write-Host "Granting Reviewer to assistant@contoso.com for $mb"
  Add-MailboxFolderPermission -Identity "$mb:\Calendar" -User "assistant@contoso.com" -AccessRights Reviewer
}

4) Verify permissions

Get-MailboxFolderPermission -Identity "alice@contoso.com:\Calendar" | Format-Table

5) Remove permissions

Remove-MailboxFolderPermission -Identity "alice@contoso.com:\Calendar" -User "assistant@contoso.com"

6) Notes for automation

  • Use service principals or managed identities where possible; avoid storing admin creds in scripts.
  • Wrap bulk operations with try/catch and log failures to a CSV for retry.
